# Git 커밋 및 푸시

## 실행 조건

- `커밋`, `푸시`, `배포`, `릴리스`를 사용자가 명시한 경우에만 수행한다.
- 일반 번역 중에는 변경을 로컬에 누적한다.

## 커밋 및 푸시 절차

1. `git status --short --branch`로 누적된 변경 파일과 현재 브랜치를 확인한다.
2. README의 번역 수치와 변경 이력이 필요한지 확인한다.
3. 관련 파일만 스테이징해 하나의 의미 있는 커밋으로 만든다.
4. 현재 브랜치를 원격 저장소에 푸시한다.
5. `git status --short --branch`로 로컬과 원격의 동기화 상태를 확인한다.

## 검사 범위

- 일반 커밋 및 푸시 요청에서는 `git status --short --branch`만 검사한다.
- 별도의 diff 검사, 번역 검사, 빌드, 설치 시험은 실행하지 않는다.
- Release 요청에서도 추가 검사를 수행하지 않고 `git status --short --branch`만 확인한다.
- 사용자가 검증 자체를 별도로 명시한 경우에만 요청한 검사를 수행한다.

## Release 절차

1. 새 버전 번호를 정한다.
2. README와 설치기 프로젝트 버전을 동일하게 갱신한다.
3. `git status --short --branch`만 확인한다.
4. 커밋과 푸시를 완료한다.
5. 새 태그를 만들고 푸시한다.
6. GitHub Actions의 `Release Installer` 성공 여부를 확인한다.
7. Release에 `SunlessSkies-KoreanPatch.zip`이 등록됐는지 확인한다.

## 규칙

- 이미 공개된 태그를 이동하거나 강제 덮어쓰지 않는다.
- 번역 몇 건만으로 새 Release를 만들지 않는다.
- 변경 이력에는 실제로 수행한 번역, 구조 변경 및 검증만 기록한다.
- Release 설명은 README의 해당 버전 변경 이력과 일치시킨다.
