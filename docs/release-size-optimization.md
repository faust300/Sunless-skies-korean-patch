# Release 패키지 용량 분석 및 축소 방안

작성일: 2026-07-12

## 목적

현재 약 85MB인 `SunlessSkies-KoreanPatch.zip`의 용량 원인을 정리하고, 설치 편의성과 안정성을 유지하면서 배포 크기를 줄일 방법을 결정한다.

## 현재 배포 구조

GitHub Actions의 `Release Installer` 워크플로는 다음 항목을 ZIP으로 묶는다.

- .NET 10 런타임을 포함한 self-contained 단일 설치기
- 번역 TXT
- 한국어 글꼴
- 패치가 완료된 Unity 에셋 파일 23개
- 패치된 `global-metadata.dat`

로컬 `payload`를 측정한 결과는 다음과 같다.

| 항목 | 압축 전 크기 |
| --- | ---: |
| `payload/Sunless Skies_Data` | 201.95MB |
| `resources.assets` | 60.44MB |
| `sharedassets0.assets` | 66.55MB |
| `global-metadata.dat` | 12.38MB |
| `payload/font` | 10.31MB |
| 나머지 `sharedassets*` 및 `level*` | 약 62.58MB |

따라서 Release 용량의 주원인은 설치기 실행 파일이 아니라, 패치된 게임 데이터 파일 전체를 배포하는 방식이다. 설치기만 압축하거나 .NET 게시 옵션만 조정해서는 큰 폭으로 줄이기 어렵다.

### 동일 파일 제외 실측

현재 Steam 설치본과 `payload`를 SHA-256으로 비교한 결과, 다음 6개 파일은 바이트 단위로 동일하다.

- `resources.assets`
- `sharedassets0.assets`
- `level4`
- `level5`
- `level16`
- `level24`

이 파일들을 제외하고 Unity 데이터만 같은 ZIP 설정으로 다시 측정한 결과는 다음과 같다.

| 구성 | ZIP 크기 |
| --- | ---: |
| 기존 23개 파일 | 44,888,013바이트 |
| 변경된 17개 파일만 | 11,286,174바이트 |
| 절감 | 33,601,839바이트 (74.86%) |

설치기는 배포에 없는 완성 파일을 건너뛰고 게임 원본을 유지하므로, 동일한 6개 파일을 Release에서 제외해도 설치 결과는 바뀌지 않는다. Release 대상은 `payload/release-static-assets.txt`에서 명시적으로 관리한다.

## 축소 방안 비교

### 1. 바이너리 델타 패치

원본 게임 파일과 패치된 파일의 차이만 `xdelta3` 또는 `bsdiff` 형식으로 생성해 배포한다. 설치기는 사용자 게임 파일의 해시를 확인한 뒤 델타를 적용하고, 결과 파일의 해시를 다시 검증한다.

장점:

- 현재 설치 방식을 크게 바꾸지 않고도 높은 압축 효과를 기대할 수 있다.
- 완성된 패치 파일을 그대로 설치하므로 현재 결과와 동일하게 유지하기 쉽다.
- 구현 범위와 위험이 설치 시 Unity 오브젝트를 직접 재작성하는 방식보다 작다.

단점:

- 게임 업데이트로 원본 파일이 변경되면 기존 델타를 적용할 수 없다.
- 지원하는 게임 버전별 원본 및 결과 해시를 관리해야 한다.
- 설치기에 델타 적용 도구를 포함하거나 델타 알고리즘을 구현해야 한다.

변경된 데이터의 분포에 따라 다르지만, 현재 약 85MB인 ZIP을 대략 10~30MB 수준으로 줄일 가능성이 있다. 실제 목표치는 델타 파일을 생성한 뒤 측정해서 확정한다.

### 2. 설치 시 Unity 오브젝트 직접 패치

완성된 `.assets` 및 `level*` 파일 대신 변경할 문자열, 오브젝트 데이터, Path ID 매핑만 배포하고 설치기가 원본 파일을 직접 수정한다.

현재 설치기는 이미 `AssetsTools.NET`을 이용해 `resources.assets`의 문자열과 TMP 폰트 오브젝트를 수정하는 코드를 포함하고 있다. 이 구조를 `sharedassets*`와 `level*`까지 확장할 수 있다.

장점:

- 배포 크기를 가장 작게 만들 수 있다.
- 번역 TXT, 글꼴 데이터, 변경 오브젝트만 배포할 수 있다.
- 작은 번역 변경마다 대형 바이너리 파일을 다시 배포하지 않아도 된다.

단점:

- Unity 직렬화 파일마다 안전한 수정 로직과 검증이 필요하다.
- 파일 내부 구조나 Path ID가 달라지면 설치 실패 또는 게임 로딩 문제가 발생할 수 있다.
- 구현과 회귀 테스트 범위가 가장 크다.

이 방식은 장기 목표로 적합하지만, 첫 용량 최적화 단계로 바로 적용하기에는 위험이 크다.

### 3. .NET 설치기 크기 최적화

현재 워크플로는 다음 조건으로 설치기를 게시한다.

- `--self-contained true`
- `PublishSingleFile=true`
- `IncludeNativeLibrariesForSelfExtract=true`

시험 가능한 옵션은 다음과 같다.

```text
-p:EnableCompressionInSingleFile=true
-p:PublishTrimmed=true
-p:TrimMode=partial
```

`EnableCompressionInSingleFile`은 EXE 자체를 줄이지만, Release ZIP에서 다시 압축되므로 최종 ZIP 절감 효과는 제한적일 수 있다. `PublishTrimmed`는 리플렉션을 사용하는 라이브러리와 충돌할 수 있으므로 `AssetsTools.NET` 기능 전체를 검증한 뒤 사용해야 한다.

`--self-contained false`는 설치기를 크게 줄일 수 있지만 사용자 PC에 .NET 10 Runtime 설치를 요구한다. 단일 실행 파일만 내려받아 설치하는 현재 배포 목표와 맞지 않으므로 기본안으로 사용하지 않는다.

### 4. 압축 형식 변경

7-Zip 또는 Zstandard 계열 압축은 ZIP보다 작아질 수 있지만 Windows 기본 압축 해제를 사용할 수 없게 된다. 사용성 저하에 비해 절감 폭이 제한적일 수 있으므로 핵심 해결책으로 보지 않는다.

## 권장 적용 순서

1. 현재 게임 원본과 패치 파일로 `xdelta3` 및 `bsdiff` 델타를 시험 생성한다.
2. 파일별 원본 크기, ZIP 크기, 델타 크기를 표로 비교한다.
3. 원본 SHA-256과 패치 결과 SHA-256을 기록한다.
4. 설치기에 원본 버전 검사, 백업, 델타 적용, 결과 검증, 실패 시 복구를 추가한다.
5. GitHub Actions가 전체 Unity 파일 대신 델타 파일을 패키징하도록 변경한다.
6. 깨끗한 Steam 설치본을 대상으로 설치 및 게임 실행을 검증한다.
7. 이후 변경이 단순한 에셋부터 설치 시 직접 패치 방식으로 점진적으로 전환한다.

## 델타 실측 결과

변경된 17개 파일을 대상으로 공식 `xdelta3` 3.2.0과 `bsdiff4` 1.2.6을 비교했다. 모든 델타는 현재 Steam 원본에서 복원한 뒤 결과 SHA-256이 `payload` 파일과 일치하는지 검증했다.

| 구성 | 전체 크기 |
| --- | ---: |
| 완성 파일 17개 | 57,485,794바이트 |
| xdelta3 델타 | 191,855바이트 |
| bsdiff4 델타 | 55,729바이트 |

`bsdiff4`가 델타 자체는 136,126바이트 더 작다. 하지만 시험에 사용한 Windows 패키지는 Python 런타임에 의존한다. 공식 xdelta3 Windows 실행 파일은 독립 실행형이며 크기가 336,896바이트이므로, 설치기 통합안은 xdelta3를 선택한다. 델타와 실행 파일을 합쳐도 약 0.5MB이며 현재 Release보다 충분히 작다.

재측정은 `tools/experimental/MeasureBinaryDeltas.ps1`을 사용한다. 이 스크립트는 델타 생성뿐 아니라 원본에서의 복원과 결과 SHA-256 검증까지 수행한다.

### 통합 Release 실측

델타 설치기 통합 후 GitHub Actions와 같은 조건으로 self-contained 단일 설치기, 번역, 글꼴, 델타, xdelta3를 ZIP으로 조립한 결과는 다음과 같다.

| 항목 | 크기 |
| --- | ---: |
| self-contained 설치기 EXE | 73,665,486바이트 |
| xdelta 델타 17개 | 191,855바이트 |
| 최종 Release ZIP | 44,025,422바이트 |

기존 약 85MB Release와 비교하면 약 41MB가 줄었다. 전체 Unity 파일이 제거된 뒤에는 self-contained .NET 설치기와 10.31MB 글꼴이 주요 용량이므로, 다음 추가 최적화 대상은 `EnableCompressionInSingleFile`의 최종 ZIP 효과 측정이다.

### .NET 게시 옵션 실측

같은 Release 구성에서 게시 옵션별 EXE와 최종 ZIP을 비교했다.

| 게시 옵션 | EXE 크기 | 최종 ZIP 크기 | 결정 |
| --- | ---: | ---: | --- |
| 기본 self-contained | 73,665,486바이트 | 44,025,422바이트 | 비교 기준 |
| `EnableCompressionInSingleFile=true` | 37,626,730바이트 | 44,237,261바이트 | 적용하지 않음 |
| `PublishTrimmed=true`, `TrimMode=full` | 13,892,768바이트 | 18,244,779바이트 | 적용 |

단일 파일 내부 압축은 EXE 자체를 줄였지만 ZIP의 이중 압축 때문에 최종 Release가 211,839바이트 커졌다. Full trimming은 최종 ZIP을 25,780,643바이트 더 줄였다.

트리밍 안전성을 위해 델타 매니페스트 JSON은 소스 생성 컨텍스트를 사용한다. Full-trim 실행 파일로 다음 항목을 검증했다.

- 17개 델타 설치 및 결과 SHA-256 일치
- 기존 파일 17개 백업
- 실패·완료 후 임시 파일 0개
- `AssetsTools.NET`을 사용하는 `resources.assets` 문자열 및 TMP 글꼴 dry-run: 3,384개 작업, 오류 0개

따라서 Release 워크플로에는 full trimming만 적용하고 단일 파일 내부 압축은 적용하지 않는다.

## 설치기 안전 요구사항

- 원본 파일 해시가 알려진 값과 다르면 델타를 적용하지 않는다.
- 파일을 변경하기 전에 현재 설치기 규칙대로 날짜가 포함된 백업을 만든다.
- 원본 파일을 직접 덮어쓰지 않고 임시 파일에 델타를 적용한다.
- 결과 해시가 예상값과 일치할 때만 임시 파일을 원본 위치로 교체한다.
- 실패하면 임시 파일을 제거하고 원본을 유지한다.
- 지원하지 않는 게임 버전이라는 오류를 사용자에게 명확히 표시한다.

## 결론

단기 권장안은 바이너리 델타 패치다. 현재 설치 결과를 유지하면서 Release 크기를 가장 현실적으로 줄일 수 있다. .NET 게시 옵션 최적화는 보조 수단으로 시험하고, Unity 오브젝트 직접 패치는 장기 구조 개선으로 진행한다.
